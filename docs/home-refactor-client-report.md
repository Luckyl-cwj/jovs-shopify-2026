# JOVS 首页重构说明

## 1. 说明目的

这份文档只聚焦 3 个甲方更关心的维度：

- 对比当前线上首页，重构版在性能上的改善点
- 对 Shopify 场景下 SEO 的影响与边界
- 后续继续开发、维护和扩展时，规范层面的收益

当前对照基线：

- 线上首页源码：[hovs.html](d:/work/新建文件夹%20(2)/jovs-shopify-2026/hovs.html)
- 重构版首页：[home.html](d:/work/新建文件夹%20(2)/jovs-shopify-2026/home.html)

---

## 2. 核心结论

本次重构不是单纯改样式，而是把首页从“可展示页面”整理成“更适合 Shopify 长期运营的首页模板基础”。

相较当前线上页，本次重构的主要价值集中在：

- 首屏以下模块不再一次性全部初始化，前端负担更轻
- 图片、视频、轮播的加载与播放策略更可控
- 代码结构、样式规则、Swiper 接入方式更统一
- 后续继续接入 Shopify Theme Editor 或 section 配置化时，基础更稳定

需要明确的是：

- 当前线上页本身已经由 Shopify 主题层输出 SEO 基础信息
- 本次重构重点在首页展示层、交互层和维护体系
- 最终上线时，应继续沿用 Shopify 主题模板已有的 SEO 输出能力

---

## 3. 与当前线上页面的主要对比

## 3.1 页面复杂度对比

按当前本地对照统计：

- 线上页 `hovs.html`
  - `114` 个 `script`
  - `38` 个样式表引用
  - `232` 张图片
  - `17` 处 `Swiper` 初始化
  - `73` 处第三方 / Shopify 扩展脚本相关引用

- 重构页 `home.html + js/home.js`
  - `2` 个 `script`
  - `4` 个样式表引用
  - `81` 张图片
  - `11` 处 `Swiper` 初始化

这两个数字不能直接简单理解为“线上差、重构版好”，因为：

- 线上页包含 Shopify 主题壳、平台脚本、营销脚本、应用扩展脚本
- 重构页当前是首页展示层与模块层的重构版本，不包含完整生产壳

但这个对比能说明一件事：

**本次重构已经把首页自身的前端展示层逻辑收得更干净、更可控。**

这对后续正式接回 Shopify 主题是有价值的，因为页面本身越清晰，越容易和平台能力、第三方应用共存。

## 4. 性能层面的优化与价值

## 4.1 已完成的性能优化

### 1. 首屏以下 Swiper 改为按需初始化

在 [js/home.js](d:/work/新建文件夹%20(2)/jovs-shopify-2026/js/home.js) 中，首页多个非首屏轮播已经改为通过 `IntersectionObserver` 延迟初始化。

当前按可见区域附近再初始化的模块包括：

- Awards Logo
- Add Text Card
- Prohibit Select
- Shop by Concern
- User Evaluation
- Trusted Recommended
- Our Mission
- Add Blog

直接收益：

- 首次进入页面时，不再一次性创建所有轮播实例
- 首屏 JS 初始化压力更低
- 首屏以下模块不会提前抢占资源

### 2. 顶部 Banner 视频改为按可见端播放

顶部 Banner 的视频已做专门优化：

- 只播放当前激活 slide 中实际可见的视频
- 桌面端和移动端视频源分开控制
- 避免两个版本视频同时抢占资源

直接收益：

- 顶部视频的加载与播放更可控
- 避免无意义的双端视频同时工作
- 首屏视频区更稳定

### 3. 图片加载策略统一

重构版首页当前图片处理已经统一为：

- `81 / 81` 张图片显式带有 `loading="lazy"`
- `81 / 81` 张图片显式带有 `decoding="async"`

对应线上页统计：

- `232` 张图片中有 `206` 张显式使用 `loading="lazy"`
- `decoding="async"` 未形成统一策略

直接收益：

- 图片异步解码更统一
- 非首屏资源更晚加载
- 页面滚动过程中的图片加载行为更可控

### 4. 媒体与首屏稳定性问题已修复

本次还处理了若干性能相关的稳定性问题，例如：

- Swiper 初始化高度错误
- 视频封面丢失
- 进度条和轮播状态不同步
- 移动端切换时的布局抖动

这些问题虽然表面上看是交互问题，本质上也会影响用户对页面性能和稳定性的感知。

---

## 4.2 对甲方的实际价值

从业务角度看，这部分优化的价值主要体现在：

- 首屏更轻，页面进入更快
- 首屏以下模块不再提前消耗太多资源
- 首页在移动端的体感更稳定
- 后续继续加内容时，更不容易出现“越做越重”的问题

---

## 4.3 对当前线上页的进一步建议

如果后续要继续做正式性能优化，**最大的收益点不一定在首页结构本身，而是在 Shopify 主题外层和第三方 App 侧。**

因为从线上页能看到：

- Shopify 平台脚本较多
- 第三方 App 扩展较多
- 营销、统计、转化、追踪类脚本同时存在

因此建议下一阶段单独做一轮：

- Shopify App 脚本审计
- 第三方统计 / 营销脚本必要性梳理
- 生产环境 Lighthouse / Web Vitals 实测

这一步会比继续单纯改页面样式更能直接提升真实加载表现。

## 5. SEO 层面的说明

## 5.1 当前 Shopify SEO 能力应继续保留

从线上页可以确认，当前站点的 SEO 基础输出已经由 Shopify 主题层处理，包括：

- `title`
- `meta description`
- `canonical`
- `hreflang`
- `Open Graph`
- `Twitter Card`

这部分能力来自 Shopify 主题模板和线上主题壳，而不是来自单个首页 HTML。

因此本次重构在 SEO 上的正确策略不是“在重构页里重新造一套 SEO 壳”，而是：

- 保持首页展示层结构可维护
- 最终上线时继续接入现有 Shopify 主题 SEO 输出链路

这是更稳妥的做法，也更符合 Shopify 建站体系。

---

## 5.2 本次重构在 SEO 侧的正向改善

虽然本次没有把重点放在 Shopify 主题 head 层改造，但首页展示层仍然做了对 SEO 有帮助的整理：

### 1. 图片可访问性更规整

对照统计：

- 线上页：`232` 张图片中，`228` 张带 `alt`，其中 `42` 张为空
- 重构页：`81` 张图片全部显式带 `alt`，其中空 `alt` 仅保留装饰性图片

这意味着重构版在图片语义控制上更整齐，更利于后续继续规范化。

### 2. 内容结构更清晰

首页模块已经按 section 拆清楚，命名也统一为：

- `section-*`
- `swiper-*`
- `btn-swiper-*`
- `swiper-progress-bar*`

这对 SEO 的间接价值在于：

- 页面内容块更容易维护
- 后续更容易映射到 Shopify section 配置
- 更利于内容更新时保持结构稳定

### 3. 避免错误地在展示层重复 SEO 输出

在 Shopify 项目里，SEO 最怕的不是“少写一段前端代码”，而是：

- 重复输出错误的 `title`
- 重复输出错误的 `canonical`
- 在局部模板里把全站 SEO 逻辑弄乱

本次重构没有破坏 Shopify 原有 SEO 输出机制，这本身就是正确做法。

---

## 5.3 需要明确的边界

当前本地重构页 [home.html](d:/work/新建文件夹%20(2)/jovs-shopify-2026/home.html) 不是最终生产主题壳，因此像下面这些生产级 SEO 能力，仍然应由 Shopify 主题模板负责：

- 页面级 `title`
- 页面级 `description`
- `canonical`
- `hreflang`
- 结构化数据
- 主题级 header / footer 注入

也就是说：

**本次重构为 SEO 提供了更好的展示层基础，但最终 SEO 成果仍应以 Shopify 主题整合后的页面输出为准。**

## 6. 后续开发规范层面的收益

这次重构最长期的价值，其实不只是当前页面变整齐，而是首页开发方式变得更规范。

## 6.1 公共样式层已经建立

本次已经补全并整理：

- [common-2026.scss](d:/work/新建文件夹%20(2)/jovs-shopify-2026/css/common-2026.scss)
- [common-2026.css](d:/work/新建文件夹%20(2)/jovs-shopify-2026/css/common-2026.css)

已沉淀内容包括：

- 颜色 token
- 圆角 token
- 间距 token
- 标题与描述公共样式
- Swiper 公共控件样式
- 常用 utility class
- `lg / xl` 响应式 utility class

这意味着：

- 后续新模块不需要重复发明样式
- 风格更容易统一
- 修改成本更低

---

## 6.2 首页 JS 逻辑更可维护

当前 [js/home.js](d:/work/新建文件夹%20(2)/jovs-shopify-2026/js/home.js) 已经形成了更清晰的模块接入方式：

- 按 section 初始化
- 按模块绑定导航与进度条
- 按可见区域延迟初始化
- 顶部视频单独控制

相比“各模块各写一套”，现在的方式更适合后续继续开发。

直接收益：

- 更容易定位问题
- 更容易新增模块
- 更容易做回归检查

---

## 6.3 更适合继续接 Shopify Theme Editor

这次重构虽然还没有把首页全部改造成 Shopify 后台完全可配置的 section schema，但已经为下一步做了基础准备：

- 模块边界更清晰
- 内容替换路径更清楚
- 公共样式层更完整
- 交互逻辑更统一

这意味着后续如果继续推进，可以更自然地往以下方向发展：

- 标题可配置
- 描述可配置
- 图片 / 视频可配置
- 按钮链接可配置
- 模块显示与排序可配置

对甲方而言，这部分价值非常实际：

- 未来更多内容改动可以转到 Shopify 后台或标准化配置层处理
- 日常活动运营会更高效
- 不必每次改一个模块都重新大范围改代码

---

## 6.4 建议形成后续统一规范

建议把这次重构沉淀成后续首页开发规范，至少包括：

### 1. 命名规范

- `section-*`
- `swiper-*`
- `btn-swiper-*`
- `swiper-progress-bar*`

### 2. 断点规范

继续沿用当前已稳定的断点思路：

- `1600+`
- `992 以下`
- `395 / 450 以下细节收口`

### 3. 样式规范

优先使用 `common-2026.scss` 中的公共 token 和 utility class，减少重复硬编码。

### 4. 轮播规范

新接入 Swiper 时，统一要求：

- 独立导航
- 独立进度条
- 断点明确
- 非首屏优先延迟初始化

### 5. 媒体规范

- 图片统一 `loading="lazy"`
- 图片统一 `decoding="async"`
- 视频按“可见 / 非可见”策略控制加载和播放

这部分规范一旦固定，后续首页维护效率会明显提高。

## 7. 甲方可直接理解的最终价值

如果用最直接的话总结，这次重构带来的价值是：

### 对当前页面

- 首页更稳定
- 多端体验更好
- 首屏以下负担更轻

### 对 SEO

- 不破坏 Shopify 原有 SEO 体系
- 图片与内容结构更规整
- 为后续整合提供更好的展示层基础

### 对后续开发

- 代码更容易维护
- 样式更容易统一
- 新模块更容易扩展
- 更适合继续接入 Shopify 的后台配置化能力

## 8. 下一阶段建议

如果下一阶段继续推进，建议优先级如下：

1. 做一轮 Shopify App / 第三方脚本审计  
   这是当前线上页性能继续优化的最大空间之一。

2. 将成熟模块逐步接入 Shopify Theme Editor 配置化  
   让 Banner、卡片、链接、文案更适合后台管理。

3. 做生产环境 Lighthouse / Web Vitals 测试  
   输出更适合对外汇报的数据结果。

4. 把首页开发规范文档化  
   让后续改版、活动页开发、模块扩展都有统一标准。

---

## 9. 一句话总结

**相较当前线上首页，本次重构的核心价值不在于“页面换了一套写法”，而在于：在 Shopify 主题体系内，把首页做成了更轻、更稳、更规范、也更适合长期运营和继续开发的版本。**
